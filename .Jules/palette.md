## 2024-05-24 - Interactive Schedule Cards Lacked Keyboard Support
**Learning:** In `LinhaDetalhesModal.tsx`, the schedule time cards (e.g., "Próximo horário") were implemented as `div` elements with `onClick` handlers but lacked `role="button"`, `tabIndex`, and keyboard event handlers. This made them completely inaccessible to keyboard users, which is a common anti-pattern when custom UI elements are built instead of native buttons.
**Action:** When implementing custom interactive elements (like clickable cards) using `div` or `span`, always add `role="button"`, `tabIndex={0}`, an `aria-label`, and an `onKeyDown` handler to support `Enter` and `Space` keys. Additionally, ensure there is a clear visual focus state (e.g., `focus-visible:ring-2`) for sighted keyboard users.

## 2024-05-24 - Avoid Adding Interactive Roles to Static Elements
**Learning:** Initially applied the same `role="button"` and `tabIndex={0}` pattern to schedule cards in `HorariosModal.tsx` simply because they looked identical visually. However, those cards were purely informational and did not have click handlers.
**Action:** Never add interactive roles (`role="button"`) or make elements focusable (`tabIndex={0}`) if they do not perform an action. This is a severe accessibility anti-pattern that misleads screen reader users and creates dead-ends for keyboard navigation. Always verify that an element actually has an interactive behavior before adding accessibility attributes intended for interactive controls.
